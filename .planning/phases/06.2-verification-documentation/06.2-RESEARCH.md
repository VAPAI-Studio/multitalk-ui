## Common Pitfalls

### Pitfall 1: Inventing Evidence
**What goes wrong:** Stating line numbers or behaviors without verifying in the actual source code.
**Why it happens:** Summaries were written at implementation time; code may have been modified by later phases.
**How to avoid:** Always grep/read the actual source file to confirm line numbers and content before citing as evidence.
**Warning signs:** Line numbers that don't match, function names that have changed.

### Pitfall 2: Missing VOL-04 Nuance
**What goes wrong:** Saying VOL-04 was complete in Phase 02 when it was only partially complete.
**Why it happens:** Phase 02 had pagination backend but no frontend "Load more" UI. Phase 6.1 completed the frontend pagination.
**How to avoid:** Document VOL-04 as "backend pagination in Phase 02, frontend Load more in Phase 6.1" with cross-reference.
**Warning signs:** The 02-04-SUMMARY.md lists VOL-04 as verified but the REQUIREMENTS.md traceability table maps VOL-04 to Phase 6.1.

### Pitfall 3: Over-Scoping
**What goes wrong:** Adding code changes, tests, or fixes alongside documentation.
**Why it happens:** Finding issues during verification review tempts fixing them.
**How to avoid:** This phase is documentation only. Note any issues found as "Anti-Patterns Found" in the verification doc, but do not fix them.
**Warning signs:** Git diff shows changes to .tsx, .py, or .ts files.

### Pitfall 4: DWNLD-02 Wording Mismatch
**What goes wrong:** Marking DWNLD-02 as not satisfied because the requirement says "presigned S3 URLs."
**Why it happens:** The original requirement wording mentions presigned URLs, but RunPod S3 does not support them.
**How to avoid:** Document that DWNLD-02's functional goal (no backend buffering of entire file) is satisfied via streaming, with a note that presigned URLs are impossible on RunPod S3 per official docs. The requirement's spirit is met.
**Warning signs:** Marking the requirement as unsatisfied when the functional behavior is correct.
